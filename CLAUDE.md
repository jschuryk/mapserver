This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Architecture

A map tile server that serves vector tiles generated from OpenStreetMap PBF files, with an IE11-compatible browser client.

**Data flow:**
```
country.osm.pbf  →  tilemaker  →  tiles.mbtiles  →  server.js  →  browser (Leaflet + VectorGrid)
```

**Tile format:** Mapbox Vector Tiles (MVT) — gzip-compressed protobuf stored in an MBTiles SQLite file. Not all tiles are guaranteed to be gzip-compressed; `server.js` checks the magic bytes (`0x1f 0x8b`) before setting `Content-Encoding: gzip`.

**Y-coordinate convention:** MBTiles uses TMS y (origin bottom-left). Leaflet uses XYZ y (origin top-left). Conversion in `server.js`: `tmsY = 2^z - 1 - y`.

**Layers defined** (in `tilemaker/config.json` and `tilemaker/process.lua`):
- `place` — named settlements (nodes only)
- `poi` — points of interest (zoom 9+)
- `transportation` — **motorway, trunk, primary only** (zoom 3+); secondary and below excluded
- `transportation_name` — named major roads (zoom 8+)
- `water` — lakes and coastline polygons (zoom 3+)
- `waterway` — **named rivers and canals only** (zoom 8+); unnamed waterways and streams excluded

**Map layers in `public/index.html`** (render order, bottom to top):
1. CSS background (`#9ab4bc`) — ocean color
2. `landPane` (Leaflet custom pane, z=150) — Natural Earth land polygons (`public/data/land.geojson`), gray fill
3. VectorGrid tile pane (z=200) — water (blue-gray), roads (gray hierarchy)
4. Overlay pane (z=400) — province/state borders, national borders (both GeoJSON)
5. Marker pane — city/province/state text labels (L.marker + L.divIcon)

**Boundary and land GeoJSON files** in `public/data/` (committed to repo, generated by `fetch-boundaries.js`):
- `land.geojson` — Natural Earth 50m world land polygons (base layer for ocean color)
- `provinces.geojson` — Natural Earth 10m state/province boundaries, filtered to CA/US/MX
- `countries.geojson` — Natural Earth 10m national boundaries, filtered to CA/US/MX
- `pipeline.geojson` — pipeline route and node data (hand-edited; updated by Python DCS script at runtime)

To regenerate boundaries: `node fetch-boundaries.js`

**JS data wrappers** in `public/data/` (gitignored, generated from GeoJSON):
- `land.js`, `provinces.js`, `countries.js`, `pipeline.js` — each wraps GeoJSON as a JS variable
- IE11 blocks XHR from `file://` to `http://`, so data is loaded via `<script>` tag injection instead
- Script tags have no cross-protocol restriction and work from local files
- `loadScript(src, callback)` in `index.html` injects a `<script>` tag; callback fires after the variable is defined
- To regenerate all wrappers: `node make-data-js.js` (run after `fetch-boundaries.js` or when GeoJSON changes)
- `pipeline.js` is regenerated by the Python DCS script every minute at runtime — `make-data-js.js` is not needed for it

**Pipeline layer** — `public/data/pipeline.geojson` contains one LineString (route) and Point features (nodes). Node properties: `type`, `node_type` (refinery/pumping_station/main_line_valve/terminal), `name`, `pressure_kpa`, `outlet_pressure_kpa` (pumping stations only), `temperature_c`, `flow_rate_m3h`. The Python DCS integration script reads live values and writes `public/data/pipeline.js` each cycle; the browser refreshes the layer every 60 seconds via `setInterval(refreshPipeline, 60000)` without a page reload. `renderPipeline()` clears `pipelineLayer` (an `L.layerGroup`) and redraws from the updated `PIPELINE_DATA` global.

**City/state/province labels** are hardcoded in `public/index.html` (VectorGrid cannot render text):
- Canadian provinces/territories (13)
- US states (50) with zoom thresholds (small NE states at z7-8, large states at z5)
- Mexico states (31) at z6-7
- Canadian cities (~25), US cities (~30 + 50 state capitals), Mexico cities (~10)
- Labels use `L.marker` + `L.divIcon` with per-label `minZoom` thresholds

**Current zoom config:** minzoom 2, maxzoom 10. Roads hidden client-side below z6 (style returns `{weight:0}`).

## Setup

### 1. Install dependencies
```
npm install
```

### 2. Generate tiles
Use tilemaker 3.0.0 in WSL (the Windows .exe crashes consistently). Run each region separately via PowerShell `Start-Process` — the Bash tool has a 10-minute cap and WSL background jobs die with the session:

```powershell
# First region — creates fresh mbtiles (no --merge)
Start-Process wsl -ArgumentList '-e tilemaker --input /mnt/c/.../canada.osm.pbf --output /mnt/c/.../tiles.mbtiles --config /mnt/c/.../tilemaker/config.json --process /mnt/c/.../tilemaker/process.lua --threads 1' -RedirectStandardOutput 'C:\...\tilemaker-canada.log' -WindowStyle Hidden -PassThru

# Additional regions — append with --merge
Start-Process wsl -ArgumentList '-e tilemaker --input /mnt/c/.../us.osm.pbf --output /mnt/c/.../tiles.mbtiles --config /mnt/c/.../tilemaker/config.json --process /mnt/c/.../tilemaker/process.lua --threads 1 --merge' -RedirectStandardOutput 'C:\...\tilemaker-us.log' -WindowStyle Hidden -PassThru
```

**Current dataset:** Canada + US + Mexico (canada-260227.osm.pbf, us-260227.osm.pbf, mexico-260227.osm.pbf). Run in order: Canada (no --merge) → Mexico → US (both --merge). Mexico is ~10 min, US is ~80 min.

Monitor: `powershell -Command "Get-Content 'C:\...\tilemaker-canada.log' -Tail 2"`

Wait for **"Filled the tileset"** — tilemaker writes all tiles in one SQLite transaction; the table shows 0 rows until it commits. Stop the server before tilemaking to avoid SQLite file locking.

OSM PBF files: https://download.geofabrik.de/

### 3. Bump cache version and restart
After retiling, bump `?v=N` in the VectorGrid URL in `public/index.html`, then:
```
npm start
```
Open http://localhost:3000. Hard refresh (Ctrl+Shift+R) in browser.

## Configuration

| Env var   | Default          | Purpose                        |
|-----------|------------------|--------------------------------|
| `PORT`    | `3000`           | HTTP port                      |
| `MBTILES` | `./tiles.mbtiles`| Path to the MBTiles file       |

- Zoom range: edit `minzoom`/`maxzoom`/`basezoom` in `tilemaker/config.json`
- Road/waterway filtering: edit `way_function()` in `tilemaker/process.lua`
- Map styling: edit `vectorTileLayerStyles` and `roadStyle()` in `public/index.html`
- Road visibility by zoom: `roadStyle()` returns `{weight:0}` below z6 — adjust threshold there
- City/state labels: hardcoded arrays `provinces` and `cities` in `public/index.html`
- Tile cache version: bump `?v=N` in the VectorGrid URL after retiling

## Tilemaker Gotchas

**Tilemaker 3.x API** — `node_keys` must be a global table, not a function:
```lua
node_keys = { "place", "name", "amenity" }   -- correct (3.x)
function node_keys() return {...} end          -- wrong (2.x)
```
`way_keys` is not used in 3.x.

**Windows crashes** — tilemaker.exe crashes immediately on this machine. Always use WSL (Kali Linux, tilemaker 3.0.0 via apt). Use `MSYS_NO_PATHCONV=1` to prevent Git Bash from mangling `/mnt/c/...` paths if running from Git Bash.

**Memory at low basezoom** — basezoom=10 requires ~15GB+ RAM during tile writing. Use `--threads 1` to avoid OOM. Default multi-threaded mode gets SIGKILL'd. Admin boundary relations (large multipolygon geometries) are currently disabled in `process.lua` — they cause OOM even with `--threads 1`.

**Dense low-zoom tiles cause browser OOM** — before road filtering, z4-5 tiles over the US averaged 2.3MB compressed (max 22MB), causing Chrome "Aw, Snap" crashes. Fixed by limiting `process.lua` to motorway/trunk/primary only. If adding more road classes back, test browser stability at z4-6.

**Building layer** — removed from `config.json` at maxzoom=10 (buildings only meaningful at zoom 13+). If added back to config, also re-enable `Layer("building", true)` in `way_function()` in `process.lua` or tilemaker throws a Lua error on every building way.

**Long runs** — use PowerShell `Start-Process` (not WSL bash background jobs or the Bash tool). The `-PassThru | Select-Object Id` pattern captures the PID for monitoring. PowerShell background jobs (`Start-Job`) do NOT survive session restarts — start each region manually.

**Nested PowerShell scripts** — `Start-Process powershell -File script.ps1 -WindowStyle Hidden` doesn't work reliably for WSL subprocesses. Run `Start-Process wsl ...` directly from a PowerShell `-Command` string instead.

## Map Styling (ASM Consortium Guidelines)

The map follows Abnormal Situation Management (ASM) Consortium HMI guidelines: neutral grayscale base, low-saturation colors, visual hierarchy through weight rather than hue.

| Element         | Color     | Notes                                      |
|-----------------|-----------|--------------------------------------------|
| Ocean           | `#9ab4bc` | CSS background (blue-gray)                 |
| Land            | `#b8b8b8` | Natural Earth land polygon GeoJSON         |
| Water/lakes     | `#9ab4bc` | VectorGrid `water` layer (matches ocean)   |
| Waterways       | `#7a9ea8` | VectorGrid `waterway` layer                |
| Motorway        | `#a0a0a0` | weight 3.5                                 |
| Trunk           | `#b4b4b4` | weight 2.5                                 |
| Primary         | `#c4c4c4` | weight 2                                   |
| Province border | `#7a7a7a` | weight 0.75, opacity 0.6                   |
| National border | `#444444` | weight 1.5, opacity 0.7                    |

Roads are hidden entirely below z6 (client-side style returns `{weight:0}`).

## Server Performance

**Bottleneck is client-side** (protobuf parsing + Canvas2D rendering), not server-side. Server responds in 1–3ms per tile.

Server-side optimizations in `server.js`:
- SQLite opened read-only with `PRAGMA cache_size = -32768` and `PRAGMA temp_store = memory`
- In-memory tile cache (Map, 10,000 tile cap, LRU-ish eviction)
- Cache pre-warmed at startup with all z2–z8 tiles (~5,600 tiles for CA+US+MX)

Client-side options in `public/index.html`:
- `updateWhenZooming: false` — tiles only fetched after zoom stops
- `keepBuffer: 4` — pre-fetches 4 tile-widths outside the viewport
- `minZoom: 4` on VectorGrid — no tiles requested below z4

## Client Deployment (air-gapped / local file)

`index.html` can be opened directly from the filesystem (`file://`) on a client machine with no internet access. Requirements:
- Set `var SERVER = 'http://<mapserver-ip>:3000';` in `index.html`
- Copy `index.html` and the `lib/` folder to the client (no `data/` needed)
- All JS/CSS libraries are local in `public/lib/` (downloaded by `download-libs.ps1`)
- Data files (GeoJSON) are served from the map server via `<script>` tag injection — no local copies required

IE11 cross-protocol restriction: XHR from `file://` to `http://` is blocked ("Access is denied" at `xhr.open()`). All data loading uses `loadScript()` instead (script tags are exempt). The `SERVER` variable is used only for tile and data script URLs.

## IE11 Constraints

All JavaScript in `public/` must be ES5-compatible:
- No `const` or `let` (use `var`)
- No arrow functions (use `function()`)
- No template literals (use string concatenation)
- No destructuring, spread operator, or `class` syntax
- Closures in loops need IIFE pattern: `(function(x){ ... })(val)`

`es6-promise.auto.min.js` polyfill (in `public/lib/`) must load before Leaflet — VectorGrid uses Promise internally and IE11 lacks it natively.

Leaflet.VectorGrid renders via Canvas2D (not WebGL), which IE11 supports. Text labels are not supported by VectorGrid — use `L.marker` with `L.divIcon` instead.

Custom Leaflet panes (`map.createPane`) are supported in IE11.
